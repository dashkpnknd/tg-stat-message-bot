# RPA Sync Design

## Рабочая схема

API на каждый кабинет не используем как основной путь. Основной путь:

```text
VK аккаунты с сохраненной сессией
  -> робот открывает ads.vk.com
  -> обходит все кабинеты аккаунта
  -> берет статистику за вчера
  -> парсит название кабинета
  -> суммирует кабинеты одного проекта
  -> пишет итог в Google-таблицу проекта
```

## Названия кабинетов

Ожидаемый формат:

```text
Проект | Город (номер)
```

Примеры:

```text
Яблочный Спас | Орск (2)
Kingstore | Нижний Новгород (2)
Apple Center | Пермь (2)
Биржа | Омск (2)
dreamstore | Абакан (2)
Аймоби | Москва (2)
Аймоби | Москва (3)
Apple Super Store | Гатчина (2)
Repoint | Саратов
Gorilla Street | Челны (2)
```

`(номер)` игнорируется. Все кабинеты с одинаковыми `Проект + Город` суммируются.

## Нулевые кабинеты

Правило:

```text
если за день расход = 0, кабинет получает статус zero_spend только за текущую проверку
```

После этого робот все равно смотрит этот кабинет на следующем ежедневном запуске. Это нужно для новых кабинетов, которые уже созданы, но еще ждут своей очереди на открутку бюджета.

## Нераспознанные кабинеты

Если название не похоже на `Проект | Город`, робот не угадывает проект. Он пишет кабинет в список нераспознанных.

Человек потом добавляет проект в реестр или правит название кабинета.

## Служебные файлы/таблицы

`projects_registry`:

```text
project_key | project_name | city | google_spreadsheet_url | google_spreadsheet_id | sheet_name | status
```

`vk_accounts`:

```text
account_label | profile_dir | status | last_login_check | last_success_at
```

`discovered_cabinets`:

```text
account_label | cabinet_id | cabinet_name | project_key | status | last_spend | last_leads
```

## Запись в таблицы

Пишем только строки:

```text
Кол-во лидов
Цена за лид
Цена за лид с учетом бонусов
Расходы
Расходы + НДС
```

Ручные строки продаж, выручки, среднего чека и ДРР не трогаем.
